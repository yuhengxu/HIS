# 登录提示、OA 工作台已办、库存流水与用户上级选择计划（2026-06-25）

## 1. 需求分析

- 登录账号或密码错误时，前端需要明确报错提示。
- 起草 OA 提交完成后应自动跳转回工作台。
- 工作台需要保留自己审批过的记录。
- 如果多个连续审批节点是同一个处理人，应自动跳过重复节点。
- 创建/编辑用户时，上级用户需要下拉选择，支持模糊搜索。
- 库存查询增加“库存录入人”列。
- 新增库存入库/出库流水查询界面。
- 系统管理员可查看全部库存和流水；其他人只能查看自己入库形成的库存及对应流水。

## 2. 业务分析

- 登录失败是用户反馈的第一入口，必须有明确原因。
- OA 发起成功后回到工作台，便于看到“我发起的流程/待处理事项”。
- 已办记录帮助审批人追踪自己处理过的流程。
- 连续同人节点重复审批没有业务价值，应自动合并处理。
- 上级用户选择应避免手输 ID 出错。
- 库存与流水必须按录入人隔离，管理员例外。

## 3. 数据分析

- 暂不新增持久化表，沿用当前内存模型。
- 库存流水需要携带 `ownerUserId` 或使用现有 `actorUserId` 作为录入/归属人。
- 流水展示需要补充物资、仓库、录入人名称。

## 4. 权限分析

- 库存列表：`SYSTEM_ADMIN` 可看全部；其他用户只能看 `ownerUserId=actorUserId`。
- 库存流水：`SYSTEM_ADMIN` 可看全部；其他用户只能看自己录入/归属的流水。
- 用户上级下拉仍使用系统管理员用户管理权限。
- 已办记录只能查看本人处理过的任务。

## 5. AI 分析

- 本轮不新增 AI 可调用写接口。
- 库存与流水读接口仍需遵守 AI 只读边界；涉及用户归属时默认按调用者权限过滤。

## 6. 系统设计

- 后端补充已办任务视图、库存流水视图、管理员全量库存/流水读取规则。
- OA 审批流转在创建下一节点前检测连续同一处理人，自动审批跳过并继续推进。
- 前端发起 OA 成功后跳转工作台。
- 前端登录捕获错误并提示。
- 用户表单上级改成远程过滤下拉。
- 新增库存流水页面与菜单。
