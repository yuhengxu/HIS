# 系统管理员无上级发起 OA 流程修复计划（2026-06-25）

## 目标

- 具备管理员权限且没有汇报上级的顶层用户发起含“汇报上级”节点的 OA 时，不因没有上级而卡在待配置状态。
- 普通用户没有上级时仍进入配置异常，避免绕过必要审批。

## 需求分析

- 组织层级应该是明确的：除最高级别人员外，其他成员都应配置 `iam_user.report_to_user_id`。
- “最高级别人员”不通过用户名判断，而是通过“无汇报上级 + 具备管理员权限”判断。
- “汇报上级”节点对最高级别人员没有实际审批人，应自动跳过并进入后续审批节点或起草确认节点。

## 业务分析

- 人工流程：顶层管理员起草 -> 汇报上级节点自动跳过 -> 后续角色/指定人审批 -> 起草人确认 -> 结束。
- 普通成员仍必须配置上级，否则流程待配置。

## 数据分析

- 不新增表字段，不改主键、索引、外键。
- 通过现有任务记录保留自动跳过节点的已处理记录，便于审计。

## 权限分析

- 仅“无汇报上级 + 具备 IAM 管理员权限”的发起人适用“无上级自动跳过”。
- 不授予 AI 或普通用户额外审批权限。

## AI 分析

- AI 不参与 OA 发起、审批、跳过逻辑。
- MCP 白名单只读边界不变。

## 系统设计

- 在 OA 运行时解析 `SUPERVISOR` 节点时，如果发起人无上级且具备 IAM 管理员权限，自动生成并通过一个跳过任务，然后推进到下一节点。
- 如果无上级但不具备 IAM 管理员权限，保持待配置状态。
- 参考常见 OA 层级审批设计：
  - 飞书审批支持管理员灵活配置审批节点规则、审批人、审批方式与权限。
  - 钉钉审批流程也由有权限的管理员在审批后台配置并发布。
  - 汇报层级审批通常按发起人向上查找主管，主管缺失时需要明确跳过或报错规则。
