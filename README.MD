# HIS + 康养平台 · 项目宪法

> 本文档是项目的**初心与宪法**。一切规划、需求、设计、开发与 AI 辅助决策，均不得违背本文档确立的目标与约束。  
> 工作方法见 [GUIDE.md](./GUIDE.md)；正式设计见 [design/](./design/)。

---

## 序言

中国快速进入老龄化社会，医疗信息化与康养服务面临巨大需求。本项目源于一个清晰判断：**医院信息系统不应把 AI 当作外挂，康养服务不应停留在机构孤岛**——要用原生 AI 能力重塑 HIS，用平台化信息化推动康养产业升级，让优质康养服务触达家庭。

本文档记录项目创立时的原始 idea，并凝练为不可轻易违背的宪法级约束。

---

## 一、使命

**以原生 AI 医院信息系统为基座，以平台化康养服务为延伸，让医疗与康养能力触达每一个家庭。**

---

## 二、三大目标

| # | 目标 | 要义 |
|---|---|---|
| 1 | **原生 AI 的 HIS** | AI 是系统内生能力，而非事后接入的外挂；临床、护理、管理等环节均可获得智能辅助 |
| 2 | **标准化康养服务** | 以信息化推动康养产业升级，提供可规范、可复制、可评估的标准化服务 |
| 3 | **康养到家平台** | 对标美团模式：以线下业务为依托，整合本地康养机构与人力资源，连接用户，实现线上下单、线下服务的闭环 |

## 三、宪法级约束

以下条款为项目**不可偏离的底线**。若需变更，须先修订本文档并联动更新 [design/](./design/) 相关文档。

### 3.1 系统边界

| 约束 | 说明 |
|---|---|
| **双模块** | 系统由 **HIS 医院信息系统** 与 **康养服务系统** 组成 |
| **独立部署** | 两模块均可单独部署，互不强制绑定 |
| **统一账户** | 共用一套账户体系，跨模块单点登录 |
| **康养双场景** | 康养系统覆盖 **院内服务** 与 **上门服务** |

### 3.2 部署与环境

| 约束 | 说明 |
|---|---|
| **跨平台** | 支持 Windows 与 Linux 部署 |
| **双模式** | 支持本地局域网部署与云端部署 |
| **开箱即用** | 前端、后端、数据库须可快速打包，一键部署整套环境 |

### 3.3 AI 原生与安全

| 约束 | 说明 |
|---|---|
| **AI 原生设计** | 自设计之初预留 AI 大模型接入能力，所有业务功能须有对应 API、接口文档与接口索引 |
| **数据库只读** | 大模型在数据库层**仅允许查询**，禁止直接写入 |
| **接口受限** | 大模型在接口层仅能调用经授权范围内的接口 |
| **权限可审计** | 所有接口须建立说明文档、权限管理与索引清单 |

### 3.4 用户体验

| 约束 | 说明 |
|---|---|
| **双端覆盖** | 网页端 + 移动端（微信小程序及企业微信小程序） |
| **轻量优先** | 面向老旧手机与低配电脑，界面简洁美观，避免加载与渲染导致严重卡顿 |
| **高并发** | 架构与实现须充分考虑并发高峰场景 |

### 3.5 工程治理

| 约束 | 说明 |
|---|---|
| **文档驱动** | 工作顺序：**需求文档 → 设计文档 → 开发文档 → 代码**；禁止跳过文档直接开发 |
| **数据可追溯** | 所有表结构、字段、主键与索引须有说明及清单，便于开发时快速定位与复用 |
| **AI 开发约束** | 借助 AI 开发时，须以文档为边界，对 AI 形成有效约束 |

### 3.6 计划驱动执行

| 约束 | 说明 |
|---|---|
| **先计划后执行** | 每轮需求进入后，先在 `plans/` 生成或引用本轮执行计划，再按计划推进文档、代码、测试工作 |
| **计划可追溯** | 计划文件须说明目标、范围、执行顺序、涉及文档/代码、测试方式与风险 |
| **计划驱动变更** | 执行过程中如发现计划缺口，先更新计划或明确偏离原因，再继续实施 |
| **计划归档** | 结构性需求的计划文件保存在 `plans/`，文件名建议使用 `YYYYMMDD-topic.plan.md` |

## 四、技术原则

本节为实现上述约束的**方向性原则**，具体技术选型见 [design/03-architecture/](./design/03-architecture/)。

- **跨平台优先**：技术栈须支持 Windows / Linux 双环境运行
- **容器化交付**：优先采用 Docker 等方式打包，实现快速部署与迁移
- **前端轻量化**：移动端以小程序为主；网页端避免过重交互与资源加载
- **双模块独立部署**：产品层分为 **HIS 模块** 与 **康养服务模块**，各自可独立打包部署；共用统一账户体系
- **模块内模块化单体**：各模块内部优先采用模块化单体架构，禁止为技术先进而拆分微服务
- **演进式拆分**：仅当用户量、团队规模、模块边界均稳定后，才允许将单个模块演进为微服务
- **接口标准化**：统一 API 规范，预留 AI 与大模型调用通道

## 五、远期方向

以下为创立时的**探索方向**，指导产品演进，但不如第三节约束刚性；落地时须纳入 PRD 与设计文档评审。

1. **智能报表**：发挥大模型在数据分析方面的优势，快速生成各类报表，支持定制
2. **用户画像**：借助大模型为员工、患者生成动态画像，辅助护理、管理与就医决策
3. **接口生态**：全部接口建立说明、权限与索引，持续完善大模型接入能力
